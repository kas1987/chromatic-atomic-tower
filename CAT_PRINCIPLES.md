# CAT Principles

## 1. Small but strict

CAT should stay lean. A small control system with hard rules beats a large knowledge base with vague guidance.

## 2. Mission before action

Work begins only after a mission exists and validates.

## 3. BEAD before dispatch

Agents do not receive broad missions. They receive atomic BEADs.

## 4. Evidence before closeout

A claim is not evidence. A test result, diff, log, report, or review artifact is evidence.

## 5. Confidence before mutation

Before write actions, the agent must know objective, scope, allowed files, stop conditions, risk, reversibility, and validation.

## 6. Gates beat vibes

Confidence, human approval, review, promotion, and tool-budget gates override agent enthusiasm.

## 7. No hidden work

Every file change must trace to Mission ID and BEAD ID.

## 8. The Orchestrator routes, it does not wander

The Orchestrator selects, scores, dispatches, monitors, and records. It should not broadly implement unless the BEAD explicitly grants it that role.

## 9. Learning is required

Each completed mission must produce a learning or a statement that no new learning was found.

## 10. CAT serves the operator

The operator should be able to type `GO` and receive a controlled, auditable next action.
